What is ORM, JPA & Hibernate?

ORM — Object Relational Mapping A technique that maps Java classes to database tables automatically. Instead of writing SQL manually, you work with Java objects and ORM converts them to DB operations behind the scenes.
JPA — Java Persistence API JPA is a specification (a set of rules / interface) that defines how Java objects should be persisted to a relational database. It is NOT an implementation — it just defines the contract.
Hibernate Hibernate is the most popular implementation of JPA. When you use Spring Data JPA, Hibernate is used internally to do the actual database work.
Spring Data JPA A Spring module that sits on top of JPA/Hibernate and provides ready-made repository interfaces. You write almost zero boilerplate — just define an interface and Spring generates the implementation automatically.
💡 Analogy: JPA = the rules of a driving test (specification).
Hibernate = the actual car you drive (implementation).
Spring Data JPA = automatic driving mode — even the gears are handled for you.

💡 ddl-auto values:
Use create only during initial dev — it wipes data every restart.
Use update for development — preserves existing data, adds new columns.
Use validate or none in production — never let Hibernate modify prod schema.

Defining Entity Class — @Entity

Definition An Entity is a Java class that is mapped to a database table. Each instance (object) of the class represents one row in the table. Mark the class with @Entity and mark the primary key field with @Id.

Employee.java — Basic Entity

Javaimport jakarta.persistence.*;

@Entity                          // marks class as a DB table
@Table(name = "employee")      // optional: custom table name (default = class name)
public class Employee {

    @Id                          // primary key
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // auto-increment
    private int eno;

    @Column(name = "employee_name", nullable = false, length = 100)
    private String ename;        // maps to column "employee_name"

    @Column(nullable = false)
    private String dept;

    @Column
    private double salary;

    // Getters and Setters
    public int    getEno()              { return eno; }
    public void   setEno(int eno)      { this.eno = eno; }
    public String getEname()            { return ename; }
    public void   setEname(String e)    { this.ename = e; }
    public String getDept()             { return dept; }
    public void   setDept(String d)     { this.dept = d; }
    public double getSalary()           { return salary; }
    public void   setSalary(double s)  { this.salary = s; }

    @Override
    public String toString() {
        return "Employee[eno="+eno+", ename="+ename+", dept="+dept+"]";
    }
}

Entity Mapping — One-to-One

Definition One-to-One: One record in Table A corresponds to exactly one record in Table B. Example: one Employee has one Address. The foreign key is stored in one of the two tables (owning side).
Employee
────── 1:1 ──────
Address
employee table has address_id (foreign key) → points to address table

Address.java — Referenced Entity

Java@Entity
@Table(name = "address")
public class Address {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int aid;

    @Column
    private String street;

    @Column
    private String city;

    @Column
    private String pincode;

    // getters, setters...
}

Employee.java — Owning Side

Java@Entity
@Table(name = "employee")
public class Employee {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int eno;

    @Column
    private String ename;

    // @OneToOne: one Employee has one Address
    // @JoinColumn: creates "address_id" FK column in employee table
    // cascade = ALL: save/delete Employee also saves/deletes Address
    @OneToOne(cascade = CascadeType.ALL)
    @JoinColumn(name = "address_id")
    private Address address;

    // getters, setters...
}

Repository, Service & Main

Java — Repository@Repository
public interface EmployeeRepo extends JpaRepository<Employee, Integer> { }
Java — Service@Service
public class EmployeeService {

    @Autowired private EmployeeRepo repo;

    public void save(Employee e) { repo.save(e); }
}
Java — Main / CommandLineRunner@SpringBootApplication
public class App {

    public static void main(String[] args) {
        SpringApplication.run(App.class, args);
    }

    @Bean
    CommandLineRunner run(EmployeeService service) {
        return args -> {
            Address addr = new Address();
            addr.setStreet("Anna Nagar");
            addr.setCity("Chennai");
            addr.setPincode("600040");

            Employee e = new Employee();
            e.setEname("Alice");
            e.setAddress(addr);         // assign address to employee

            service.save(e);
            // Spring saves BOTH employee AND address (cascade = ALL)
        };
    }
}
💡 CascadeType.ALL means any operation on Employee (save, delete, update) is automatically cascaded to Address. If you delete Employee, Address is also deleted.

Entity Mapping — One-to-Many & Many-to-One

Definition One-to-Many: One Department has many Employees.
Many-to-One: Many Employees belong to one Department.
These are two sides of the same relationship. The FK dept_id lives in the employee table (many side).
Department (One)
────── 1:N ──────
Employee (Many)
employee table has dept_id (foreign key) → points to department table

Department.java — One Side

Java@Entity
public class Department {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int did;

    @Column
    private String dname;

    // One Department has MANY Employees
    // mappedBy = "dept" → refers to the field name in Employee class
    // cascade = ALL → save dept also saves all its employees
    // fetch = LAZY  → employees loaded only when accessed (better performance)
    @OneToMany(mappedBy = "dept", cascade = CascadeType.ALL, fetch = FetchType.LAZY)
    private List<Employee> employees = new ArrayList<>();

    // getters, setters...
}

Employee.java — Many Side (owns the FK)

Java@Entity
public class Employee {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int eno;

    @Column
    private String ename;

    // Many Employees belong to ONE Department
    // @JoinColumn: creates "dept_id" FK column in employee table
    @ManyToOne(fetch = FetchType.EAGER)
    @JoinColumn(name = "dept_id")
    private Department dept;

    // getters, setters...
}

Usage — CommandLineRunner

Java@Bean
CommandLineRunner run(DepartmentRepo deptRepo) {
    return args -> {
        Department d = new Department();
        d.setDname("Engineering");

        Employee e1 = new Employee(); e1.setEname("Alice"); e1.setDept(d);
        Employee e2 = new Employee(); e2.setEname("Bob");   e2.setDept(d);

        d.setEmployees(Arrays.asList(e1, e2));

        deptRepo.save(d);  // saves dept + e1 + e2 (cascade)
    };
}

Entity Mapping — Many-to-Many

Definition Many-to-Many: One Student can enroll in many Courses. One Course can have many Students. A join table (student_course) is created automatically to hold the relationship.
Student
──── M:N ────
Course
join table: student_course (student_id, course_id)

Student.java — Owning Side

Java@Entity
public class Student {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int sid;

    @Column
    private String sname;

    // @ManyToMany: student ↔ course
    // @JoinTable: defines the join table name and FK columns
    @ManyToMany(cascade = CascadeType.ALL, fetch = FetchType.LAZY)
    @JoinTable(
        name = "student_course",                          // join table name
        joinColumns = @JoinColumn(name = "student_id"),   // FK for Student
        inverseJoinColumns = @JoinColumn(name = "course_id") // FK for Course
    )
    private List<Course> courses = new ArrayList<>();

    // getters, setters...
}

Course.java — Inverse Side

Java@Entity
public class Course {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int cid;

    @Column
    private String cname;

    // mappedBy = "courses" → refers to field name in Student class
    // Inverse side — does not own the join table
    @ManyToMany(mappedBy = "courses")
    private List<Student> students = new ArrayList<>();

    // getters, setters...
}

Usage

Java@Bean
CommandLineRunner run(StudentRepo studentRepo) {
    return args -> {
        Course c1 = new Course(); c1.setCname("Java");
        Course c2 = new Course(); c2.setCname("Spring");

        Student s = new Student();
        s.setSname("Alice");
        s.setCourses(Arrays.asList(c1, c2));

        studentRepo.save(s);
        // Creates: student row, course rows, student_course rows
    };
}

One-to-Many / Many-to-One2 tablesFK in "many" side table

CrudRepository

Definition CrudRepository is the most basic Spring Data repository interface. It provides standard CRUD operations automatically — no implementation needed. You just declare the interface and Spring generates the implementation at runtime.

Declaring a CrudRepository

Javaimport org.springframework.data.repository.CrudRepository;

// CrudRepository<EntityClass, PrimaryKeyType>
@Repository
public interface EmployeeRepo extends CrudRepository<Employee, Integer> {
    // No code needed — Spring provides all basic CRUD methods
}

Built-in Methods from CrudRepository

Java — Service using CrudRepository@Service
public class EmployeeService {

    @Autowired private EmployeeRepo repo;

    // INSERT or UPDATE (if id exists → update, else → insert)
    public void save(Employee e)          { repo.save(e); }

    // SELECT by id — returns Optional (may or may not have value)
    public Optional<Employee> findById(int id) { return repo.findById(id); }

    // SELECT all rows
    public Iterable<Employee> findAll()  { return repo.findAll(); }

    // DELETE by id
    public void deleteById(int id)       { repo.deleteById(id); }

    // DELETE object
    public void delete(Employee e)        { repo.delete(e); }

    // COUNT all rows
    public long count()                   { return repo.count(); }

    // CHECK if id exists
    public boolean exists(int id)         { return repo.existsById(id); }

    // SAVE multiple at once
    public void saveAll(List<Employee> list) { repo.saveAll(list); }
}

JpaRepository

Definition JpaRepository extends CrudRepository and adds more powerful methods — pagination, sorting, flushing, and batch operations. This is the most commonly used repository interface in Spring Data JPA.

JpaRepository Declaration

Javaimport org.springframework.data.jpa.repository.JpaRepository;

// JpaRepository<EntityClass, PrimaryKeyType>
@Repository
public interface EmployeeRepo extends JpaRepository<Employee, Integer> {
    // Spring generates all implementation automatically
}

Extra Methods in JpaRepository (beyond CrudRepository)

Java — Service using JpaRepository@Service
public class EmployeeService {

    @Autowired private EmployeeRepo repo;

    // findAll with Sorting
    public List<Employee> findAllSorted() {
        return repo.findAll(Sort.by("ename").ascending());
    }

    // findAll with Pagination
    public Page<Employee> findByPage(int pageNo, int pageSize) {
        Pageable pageable = PageRequest.of(pageNo, pageSize);
        return repo.findAll(pageable);
    }

    // findAll with Pagination + Sorting
    public Page<Employee> findSortedPage(int pageNo, int pageSize) {
        Pageable pageable = PageRequest.of(pageNo, pageSize,
                                Sort.by("salary").descending());
        return repo.findAll(pageable);
    }

    // flush — force pending changes to DB immediately
    public void flushChanges() { repo.flush(); }

    // saveAndFlush — save + flush in one call
    public Employee saveNow(Employee e) { return repo.saveAndFlush(e); }
}

Custom Query Methods (Derived Queries)

Definition Spring Data JPA can automatically generate SQL from method names in your repository interface. Just follow the naming convention — Spring parses the method name and creates the query. 
Java — EmployeeRepo.java@Repository
public interface EmployeeRepo extends JpaRepository<Employee, Integer> {

    // SELECT * FROM employee WHERE ename = ?
    List<Employee> findByEname(String ename);

    // SELECT * FROM employee WHERE dept = ?
    List<Employee> findByDept(String dept);

    // SELECT * FROM employee WHERE salary > ?
    List<Employee> findBySalaryGreaterThan(double salary);

    // SELECT * FROM employee WHERE salary BETWEEN ? AND ?
    List<Employee> findBySalaryBetween(double min, double max);

    // SELECT * FROM employee WHERE ename LIKE '%?%'
    List<Employee> findByEnameContaining(String keyword);

    // SELECT * FROM employee WHERE dept = ? AND salary > ?
    List<Employee> findByDeptAndSalaryGreaterThan(String dept, double salary);

    // SELECT * FROM employee WHERE ename = ? OR dept = ?
    List<Employee> findByEnameOrDept(String ename, String dept);

    // SELECT * FROM employee ORDER BY salary DESC
    List<Employee> findAllByOrderBySalaryDesc();

    // SELECT COUNT(*) FROM employee WHERE dept = ?
    long countByDept(String dept);

    // DELETE FROM employee WHERE dept = ?
    void deleteByDept(String dept);
}

JPQL — Java Persistence Query Language

Definition JPQL is a query language similar to SQL, but instead of table/column names, you use Java class and field names. Hibernate translates JPQL to the actual SQL for your database. Use @Query annotation to write JPQL in your repository.
💡 JPQL vs SQL:
SQL: SELECT * FROM employee WHERE dept = 'IT'
JPQL: SELECT e FROM Employee e WHERE e.dept = 'IT'
JPQL uses class name Employee (not table name) and field name e.dept (not column name). 
Java — EmployeeRepo.java with @Query (JPQL)@Repository
public interface EmployeeRepo extends JpaRepository<Employee, Integer> {

    // Basic JPQL — SELECT all by department
    @Query("SELECT e FROM Employee e WHERE e.dept = :dept")
    List<Employee> findByDeptJPQL(@Param("dept") String dept);

    // JPQL with multiple conditions
    @Query("SELECT e FROM Employee e WHERE e.dept = :dept AND e.salary > :sal")
    List<Employee> findByDeptAndSalary(@Param("dept") String dept,
                                       @Param("sal") double sal);

    // JPQL with ORDER BY
    @Query("SELECT e FROM Employee e ORDER BY e.salary DESC")
    List<Employee> findAllOrderedBySalary();

    // JPQL — SELECT specific fields only
    @Query("SELECT e.ename, e.dept FROM Employee e WHERE e.salary > :sal")
    List<Object[]> findNamesAndDepts(@Param("sal") double sal);

    // JPQL — COUNT
    @Query("SELECT COUNT(e) FROM Employee e WHERE e.dept = :dept")
    long countByDeptJPQL(@Param("dept") String dept);

    // JPQL — UPDATE (must use @Modifying + @Transactional)
    @Modifying
    @Transactional
    @Query("UPDATE Employee e SET e.salary = e.salary * 1.1 WHERE e.dept = :dept")
    int incrementSalary(@Param("dept") String dept);

    // JPQL — DELETE
    @Modifying
    @Transactional
    @Query("DELETE FROM Employee e WHERE e.dept = :dept")
    void deleteByDeptJPQL(@Param("dept") String dept);

    // Native SQL Query — use actual table/column names
    @Query(value = "SELECT * FROM employee WHERE dept = :dept",
           nativeQuery = true)
    List<Employee> findByDeptNative(@Param("dept") String dept);
}

Criteria Query

Definition Criteria Query is a programmatic, type-safe way to build queries using Java code instead of query strings. It is part of the JPA specification and uses CriteriaBuilder and CriteriaQuery API. Useful when queries are built dynamically at runtime.
💡 When to use Criteria Query?
When the query conditions are unknown until runtime — e.g., a search filter form where the user may or may not fill in dept, salary, ename. With JPQL you'd need multiple queries; with Criteria API you build one query dynamically.

EmployeeService.java — Criteria Query Examples

Java@Service
public class EmployeeService {

    @PersistenceContext
    private EntityManager em;  // JPA's main interface for DB operations

    // ── Find all employees ──────────────────────────────────────────────
    public List<Employee> findAll() {
        CriteriaBuilder   cb = em.getCriteriaBuilder();
        CriteriaQuery<Employee> cq = cb.createQuery(Employee.class);
        Root<Employee>    root = cq.from(Employee.class);  // FROM employee

        cq.select(root);   // SELECT *

        return em.createQuery(cq).getResultList();
    }

    // ── Find by department ──────────────────────────────────────────────
    public List<Employee> findByDept(String dept) {
        CriteriaBuilder   cb   = em.getCriteriaBuilder();
        CriteriaQuery<Employee> cq = cb.createQuery(Employee.class);
        Root<Employee>    root = cq.from(Employee.class);

        // WHERE e.dept = :dept
        Predicate deptPredicate = cb.equal(root.get("dept"), dept);
        cq.select(root).where(deptPredicate);

        return em.createQuery(cq).getResultList();
    }

    // ── Dynamic query — multiple optional filters ─────────────────────
    public List<Employee> search(String dept, Double minSalary, String ename) {
        CriteriaBuilder   cb   = em.getCriteriaBuilder();
        CriteriaQuery<Employee> cq = cb.createQuery(Employee.class);
        Root<Employee>    root = cq.from(Employee.class);

        List<Predicate> predicates = new ArrayList<>();

        if (dept != null)
            predicates.add(cb.equal(root.get("dept"), dept));

        if (minSalary != null)
            predicates.add(cb.greaterThan(root.get("salary"), minSalary));

        if (ename != null)
            predicates.add(cb.like(root.get("ename"), "%" + ename + "%"));

        cq.select(root).where(cb.and(predicates.toArray(new Predicate[0])));

        return em.createQuery(cq).getResultList();
    }

    // ── ORDER BY salary DESC ─────────────────────────────────────────
    public List<Employee> findAllOrderedBySalary() {
        CriteriaBuilder   cb   = em.getCriteriaBuilder();
        CriteriaQuery<Employee> cq = cb.createQuery(Employee.class);
        Root<Employee>    root = cq.from(Employee.class);

        cq.select(root).orderBy(cb.desc(root.get("salary")));

        return em.createQuery(cq).getResultList();
    }
}
CriteriaBuilder MethodSQL Equivalent

Spring Data JPA & Transactions

Definition Spring Data JPA integrates with Spring's transaction management. Repository methods like save() and delete() are already transactional by default. For custom business logic involving multiple operations, use @Transactional on service methods.
💡 Default behaviour: All Spring Data JPA repository methods (save, delete, findAll etc.) are already wrapped in transactions automatically — you don't need to add @Transactional on them. You only add it on your own service methods that do multiple operations.

Setup in application.properties

application.properties# Spring Boot auto-configures JpaTransactionManager — no extra setup needed!
spring.datasource.url=jdbc:mysql://localhost:3306/empdb
spring.datasource.username=root
spring.datasource.password=root
spring.jpa.hibernate.ddl-auto=update

Service — @Transactional on Multi-Step Operations

Java — EmployeeService.java@Service
public class EmployeeService {

    @Autowired private EmployeeRepo  empRepo;
    @Autowired private DepartmentRepo deptRepo;

    // ── Single operation — already transactional via JPA ────────────────
    public void insert(Employee e) {
        empRepo.save(e);   // no @Transactional needed — save() handles it
    }

    // ── Multi-step operation — needs @Transactional ──────────────────────
    @Transactional
    public void transferEmployee(int empId, int newDeptId) {
        Employee e    = empRepo.findById(empId)
                               .orElseThrow(() -> new RuntimeException("Not found"));
        Department d  = deptRepo.findById(newDeptId)
                                .orElseThrow(() -> new RuntimeException("Not found"));
        e.setDept(d.getDname());
        empRepo.save(e);
        // if exception thrown here → empRepo.save() is rolled back
    }

    // ── Read-only transaction — optimization ──────────────────────────────
    @Transactional(readOnly = true)
    public List<Employee> getAll() {
        return empRepo.findAll();
    }

    // ── Rollback on checked exception ────────────────────────────────────
    @Transactional(rollbackFor = Exception.class)
    public void complexOperation(Employee e) throws Exception {
        empRepo.save(e);
        if (e.getSalary() < 0)
            throw new Exception("Invalid salary");  // triggers rollback
    }

    // ── @Transactional with JPQL @Modifying ──────────────────────────────
    @Transactional
    public void giveRaise(String dept) {
        empRepo.incrementSalary(dept);  // calls @Modifying @Query method
    }
}
⚠ Important rules:
1. Always put @Transactional on service layer methods — not on repository or controller.
2. @Modifying queries in repository MUST have @Transactional.
3. Spring only rolls back on RuntimeException by default. Use rollbackFor=Exception.class for checked exceptions.

Interview Questions & Answers

Q1. What is the difference between JPA, Hibernate, and Spring Data JPA?

Answer JPA is a specification (interface/rules). Hibernate is the implementation of JPA that does the actual work. Spring Data JPA is a Spring module that builds on top of JPA/Hibernate and provides ready-made repository interfaces, eliminating most boilerplate. You use Spring Data JPA → it delegates to JPA → Hibernate executes the SQL.

Q2. What is @Entity and what are the mandatory requirements for an Entity class?

Answer @Entity marks a Java class as a DB table. Mandatory requirements: (1) @Entity annotation on the class, (2) @Id annotation on the primary key field, (3) a no-argument constructor (can be default). Optional: @Table to set custom table name, @Column to customize column properties.

Q3. What is ddl-auto and what are its values?

Answer ddl-auto controls how Hibernate manages DB schema. Values: create (drop + recreate tables each startup — wipes data), update (add new columns/tables, keep existing data — for dev), validate (check schema matches entity, no changes), none (do nothing — for prod). Always use validate or none in production.

Q4. What is the difference between CrudRepository and JpaRepository?

Answer CrudRepository provides basic CRUD methods, findAll() returns Iterable. JpaRepository extends CrudRepository and adds pagination (findAll(Pageable)), sorting (findAll(Sort)), flush(), saveAndFlush(), batch operations, and findAll() returns List. JpaRepository is recommended in most projects.

Q5. What are derived query methods in Spring Data JPA?

Answer Derived queries are repository methods whose SQL is generated automatically by Spring based on the method name. For example, findByEname(String ename) generates SELECT * FROM employee WHERE ename=?. Keywords like And, Or, GreaterThan, Between, Containing, OrderBy are supported. For complex queries, use @Query with JPQL.

Q6. What is JPQL? How is it different from SQL?

Answer JPQL (Java Persistence Query Language) uses Java class and field names instead of DB table and column names. SQL: SELECT * FROM employee WHERE dept='IT'. JPQL: SELECT e FROM Employee e WHERE e.dept='IT'. JPQL is DB-independent — Hibernate translates it to the correct SQL for the target database. Use @Query on repository methods to write JPQL.

Q7. What is @Modifying and when is it required?

Answer @Modifying is required on @Query methods that perform INSERT, UPDATE, or DELETE operations (not SELECT). It tells Spring Data JPA that this query modifies the database. It must be used together with @Transactional. Without @Modifying, Spring will throw an InvalidDataAccessApiUsageException.

Q8. What is the difference between One-to-One, One-to-Many, and Many-to-Many?

Answer One-to-One: one Employee has one Address — FK in one table. One-to-Many: one Department has many Employees — FK (dept_id) in the Employee table (many side). Many-to-Many: one Student can have many Courses and vice versa — requires a join table (student_course) holding both FKs. The @JoinTable annotation defines this join table.

Q9. What is mappedBy in @OneToMany?

Answer mappedBy tells JPA which side owns the relationship (holds the FK). The side with mappedBy is the inverse side — it does not own the FK column. For example, @OneToMany(mappedBy="dept") in Department means the dept field in Employee class owns the FK. Without mappedBy, JPA would create an unnecessary extra join table.

Q10. What is FetchType.LAZY vs EAGER?

Answer LAZY: related data is loaded only when the field is actually accessed in code — better performance, avoids loading unnecessary data. EAGER: related data is loaded immediately when the parent is loaded — simpler but can cause performance issues. Default: @OneToMany and @ManyToMany are LAZY. @ManyToOne and @OneToOne are EAGER.

Q11. What is CascadeType? Explain CascadeType.ALL.

Answer CascadeType defines which JPA operations on the parent entity should cascade to the child entity. CascadeType.ALL means ALL operations (save, update, delete, refresh, merge, detach) on the parent are automatically applied to the child. Example: if Employee has cascade=ALL on Address, saving Employee also saves Address; deleting Employee also deletes Address.

Q12. What is Criteria Query? When would you use it over JPQL?

Answer Criteria Query is a programmatic type-safe way to build queries using Java API (CriteriaBuilder, CriteriaQuery, Root, Predicate). Use it when query conditions are not known until runtime — e.g., a dynamic search form where the user may or may not fill in dept, salary, or name. JPQL requires the full query string upfront; Criteria API builds conditions dynamically in code.

Q13. When do you need @Transactional in Spring Data JPA?

Answer JPA repository methods (save, delete, findAll) are already transactional — no @Transactional needed on them. You need @Transactional on service methods when: (1) doing multiple DB operations that must succeed or fail together, (2) using @Modifying @Query methods, (3) doing read-heavy operations (readOnly=true for optimization). Always place @Transactional on the service layer, not repository or controller.
